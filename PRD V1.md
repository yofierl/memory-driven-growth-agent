# Memory-Driven Growth Agent PRD V1

## 1. 产品名称

**Memory-Driven Growth Agent**

中文名暂定：

**记忆驱动成长教练 Agent**

---

## 2. 产品定位

Memory-Driven Growth Agent 是一个面向长期成长陪伴场景的 AI Agent 系统。

它不是心理咨询师替代品，也不是普通情绪聊天机器人，而是一个通过长期记忆、主动信息采集、行为模式发现和干预策略推荐，帮助用户从「自我觉察」走向「实际行动」的成长教练系统。

系统核心目标是解决传统 AI 助手在长期陪伴场景中的三个问题：

1. **记不住用户长期经历**
2. **无法从多轮对话中发现重复行为模式**
3. **只能安慰用户，难以推动用户采取行动**

---

## 3. 项目目标

### 3.1 第一目标

该项目首先作为 **AI Agent / LLM 应用开发方向的作品集项目**，用于秋招、实习面试展示。

项目需要体现：

* 长期记忆系统设计能力
* LangGraph Agent 工作流编排能力
* 用户画像与行为模式分析能力
* 干预策略路由与行动跟踪能力
* 从产品问题到系统设计的完整思考能力

### 3.2 次要目标

该项目也希望具备一定个人使用价值，能够帮助用户缓解内耗、焦虑、高敏感、拖延等问题，并辅助用户进行长期成长记录。

### 3.3 非目标

本项目不做以下内容：

* 不做心理疾病诊断
* 不做抑郁症、焦虑症等医学评估
* 不替代心理咨询师
* 不提供医疗建议
* 不处理严重自伤、自杀、暴力风险场景的专业干预
* 不做复杂商业化系统
* 不做大型社区产品

---

## 4. 目标用户

### 4.1 核心用户

有长期成长需求的年轻用户，尤其是：

* 容易内耗的人
* 经常焦虑的人
* 高敏感人群
* 容易拖延的人
* 面临学习、求职、项目、职业选择压力的人
* 希望通过 AI 进行自我觉察和行动管理的人

### 4.2 典型用户画像

#### 用户 A：求职焦虑型学生

* 正在准备实习或秋招
* 经常因为项目进度、简历、面试表现焦虑
* 知道应该学习，但容易拖延
* 希望有人帮自己梳理问题、制定行动计划并持续跟踪

#### 用户 B：高敏感内耗型用户

* 容易受到他人评价影响
* 经常因为一句话、一个回复、一个态度陷入情绪波动
* 希望减少对外界反馈的依赖
* 希望建立更稳定的自我评价体系

#### 用户 C：长期迷茫型用户

* 对未来方向不清楚
* 经常思考人生意义、赚钱目的、职业选择
* 希望通过结构化问题逐渐明确人生目标

## 5. 核心价值主张

### 5.1 一句话价值主张

> 一个能够记住用户长期经历、发现重复行为模式，并推动用户采取行动的成长教练 Agent。

### 5.2 对用户的价值

用户使用该系统后，应感受到：

1. **它记得我**

   * 系统能记住用户过去的重要经历、情绪触发点、行为反应和成长目标。

2. **它看见了我的模式**

   * 系统能从长期对话中发现用户反复出现的问题链路，例如：

     * 任务压力 → 焦虑 → 刷视频 → 自责
     * 他人评价 → 情绪波动 → 反复回想 → 学习中断

3. **它能推动我行动**

   * 系统不是单纯安慰，而是帮助用户拆解任务、选择方法、跟踪执行、复盘结果。

### 5.3 对面试展示的价值

项目要体现的核心亮点排序为：

1. **多层记忆系统**
2. **Agent 工作流编排**
3. **成长方法论落地**
4. **数据治理与用户控制**

## 6. 产品核心理念

本项目采用「自由对话 + 长期观察 + 成长计划」的混合模式。

### 6.1 不采用纯自由聊天

纯自由聊天容易退化为：

```text
用户输入
↓
LLM 回复
```

这与普通 ChatGPT、豆包、Claude 区别不明显。

### 6.2 不采用纯打卡训练营

纯打卡训练营容易退化为习惯养成 App，用户长期坚持成本高。

### 6.3 采用混合模式

系统默认以自由对话开始。

当 Agent 发现用户存在持续性问题，例如：

* 长期焦虑
* 长期内耗
* 长期拖延
* 反复受到某类人际关系影响
* 反复因为任务压力逃避行动

系统会主动建议用户进入成长计划或执行某个干预方法。

核心流程：

```text
自由对话
↓
主动信息采集
↓
长期记忆沉淀
↓
行为模式发现
↓
成长方案推荐
↓
行动执行跟踪
↓
复盘与记忆更新
```

## 7. MVP 范围

### 7.1 MVP 目标

MVP 阶段不要求证明用户一定能够改变，也不追求完整成长管理系统。

MVP 阶段只需要证明一件事：

> Agent 能够把自然对话中的信息沉淀为可追溯记忆，并基于重复证据生成一个可执行行动任务。

最小闭环为：

```text
用户表达困扰
↓
抽取结构化记忆
↓
召回相关历史记忆
↓
发现有证据支持的重复模式
↓
匹配一个低难度干预方法
↓
生成一个轻量行动任务
↓
记录任务反馈
```

MVP 不承诺用户焦虑、拖延或内耗显著改善，只验证系统链路是否成立、输出是否可信、用户是否可以校正记忆和模式。

### 7.2 MVP 需要实现的能力

P0 MVP 只包含以下能力：

1. 用户自由对话
2. 情绪事件结构化抽取
3. 信息缺失检测与单问题追问
4. 长期记忆存储
5. 历史记忆召回
6. 基于证据的简单行为模式发现
7. 干预方法匹配
8. 行动任务生成
9. 任务完成情况记录
10. 记忆和模式的用户校正入口

### 7.3 MVP 暂不实现的内容

以下能力进入 P1 或后续版本，不作为第一版完成标准：

1. 18 天成长计划
2. 完整周总结页面
3. Growth Story 成长轨迹
4. 复杂用户画像动态演化
5. Redis 会话缓存与限流
6. 多会话管理
7. 成长报告导出
8. 多端提醒和移动端体验

### 7.4 MVP 用户故事与验收场景

#### 场景一：学习拖延模式

作为求职焦虑型学生，当我连续多次提到「学不进去」「任务太大」「刷视频逃避」时，系统应能：

1. 抽取每次对话中的事件、情绪、触发因素、行为和结果
2. 在至少 3 条证据记忆后生成候选模式
3. 引用 evidence_memory_ids 说明模式来源
4. 询问我是否认可该模式
5. 在我认可后推荐「15 分钟启动法」
6. 生成一个 15 到 30 分钟内可完成的小任务

#### 场景二：信息不足时轻量追问

当我只说「最近好焦虑」时，系统不应直接下结论或生成宏大建议，而应先承接情绪，并只追问一个关键缺失信息，例如最近是否有明确触发事件。

#### 场景三：用户校正记忆

当系统保存的记忆不准确时，我可以查看、修改或删除该记忆。被删除的记忆不应继续参与召回、模式发现或用户画像更新。

#### 场景四：用户拒绝候选模式

当系统发现一个候选模式并询问我是否认可时，如果我选择「不准确」或「暂时忽略」，系统应将该模式标记为 rejected 或 ignored，不再基于该模式推荐干预方法、生成行动任务或更新用户画像。

## 8. 核心功能模块

## 8.1 对话模块

### 功能说明

用户可以像普通聊天一样输入当前困扰、情绪、问题或想法。

Agent 需要先承接用户表达，再根据上下文决定是否追问、分析、召回记忆或推荐行动。

### 用户输入示例

```text
最近状态不太好。
```

```text
我今天又没学进去，感觉自己很废。
```

```text
我总是很在意别人怎么看我。
```

```text
我知道应该学习，但就是不想开始。
```

### Agent 回复原则

Agent 回复需要遵循：

1. 先理解和承接，不直接说教
2. 信息不足时进行轻量追问
3. 不一次问太多问题
4. 不强行诊断用户
5. 不夸大心理问题
6. 尽量将问题引导到可执行行动

---

## 8.2 主动记忆采集模块

### 功能说明

用户的自然表达通常是不完整的。

例如用户只说：

```text
最近好焦虑。
```

系统无法直接知道：

* 因为什么焦虑
* 焦虑持续多久
* 焦虑后做了什么
* 造成了什么影响

因此需要设计主动记忆采集机制。

### 核心能力

系统需要识别用户表达中缺失的信息字段，并通过自然追问补全。

### 记忆字段

每个情绪事件尽量抽取以下字段：

```json
{
  "event": "发生了什么",
  "emotion": "用户的情绪",
  "trigger": "触发因素",
  "behavior": "用户的应对行为",
  "result": "造成的结果",
  "time": "发生时间",
  "scenario": "所属场景",
  "confidence": "置信度"
}
```

### 示例

用户输入：

```text
最近好焦虑。
```

系统识别：

```json
{
  "emotion": "焦虑",
  "event": null,
  "trigger": null,
  "behavior": null,
  "result": null
}
```

Agent 追问：

```text
这种焦虑最近有没有一个特别明显的触发点？比如学习、项目、求职、人际关系，或者某件具体的事？
```

### 追问策略

追问必须满足：

1. 一次只问一个问题
2. 优先补全最关键缺失字段
3. 追问不能像问卷
4. 用户不想回答时不强迫
5. 追问后需要更新记忆状态

---

## 8.3 多层记忆系统

### 功能说明

系统需要将用户信息分层存储，而不是简单保存完整聊天记录。

### 记忆层级

完整架构为五层记忆。MVP 先实现 Short-Term Memory、Summary Memory、Long-Term Memory 和 User Profile，Growth Story 作为 P1 能力后置。

```text
Short-Term Memory
Summary Memory
Long-Term Memory
User Profile
Growth Story
```

---

### 8.3.1 Short-Term Memory

短期记忆保存最近几轮对话，用于保证当前对话连贯性。

内容包括：

* 最近用户输入
* 最近 Agent 回复
* 当前对话状态
* 当前正在追问的问题
* 当前正在执行的行动任务

---

### 8.3.2 Summary Memory

摘要记忆保存阶段性对话总结。

适合在多轮对话后生成，例如每 5 轮或每次会话结束后生成。

内容包括：

* 本轮主要问题
* 关键情绪
* 触发因素
* 用户态度
* 暂定行动建议

---

### 8.3.3 Long-Term Memory

长期记忆保存重要事件、长期问题和重复模式。

内容包括：

* 重要成长事件
* 反复出现的压力源
* 典型情绪反应
* 典型回避行为
* 重要人生目标
* 长期执行计划

示例：

```json
{
  "type": "emotion_event",
  "scenario": "秋招准备",
  "event": "准备面试时学不进去",
  "emotion": "焦虑",
  "trigger": "任务压力过大",
  "behavior": "刷视频回避",
  "result": "学习中断并产生自责",
  "importance": 4,
  "confidence": 0.82
}
```

---

### 8.3.4 User Profile

用户画像保存相对稳定的用户特征。

内容包括：

* 用户长期目标
* 用户常见压力源
* 用户常见情绪模式
* 用户常见行为模式
* 用户偏好的鼓励方式
* 用户不喜欢的建议方式

示例：

```json
{
  "traits": ["容易焦虑", "对外界评价敏感", "任务过大时容易拖延"],
  "goals": ["找到 AI 应用开发实习", "提升项目能力"],
  "preferred_support_style": "具体、直接、可执行"
}
```

---

### 8.3.5 Growth Story（P1）

成长轨迹记录用户阶段性变化。MVP 不实现自动生成，只保留为后续架构设计。

内容包括：

* 阶段目标
* 阶段困难
* 阶段行动
* 阶段复盘
* 阶段变化

示例：

```text
用户在秋招准备阶段反复出现项目焦虑和学习拖延。
早期主要表现为任务压力过大后刷视频回避。
后续通过 15 分钟启动法开始尝试降低启动难度。
```

---

## 8.4 记忆召回模块

### 功能说明

当用户提出新问题时，系统需要从长期记忆中召回相关历史信息，生成更个性化的回复。

### 召回依据

可以根据以下维度召回：

* 情绪相似
* 场景相似
* 触发因素相似
* 行为模式相似
* 当前目标相关
* 当前成长计划相关

### 示例

用户输入：

```text
我今天又学不进去了。
```

系统召回：

```text
用户之前多次提到，在秋招准备和项目推进时，任务压力过大会导致焦虑，并通过刷视频逃避，之后产生自责。
```

Agent 回复：

```text
你之前也提到过，任务一大时容易进入“焦虑—回避—自责”的循环。今天先不要要求自己学完整块内容，我们只把目标降到一个 15 分钟的小任务。
```

---

## 8.5 行为模式发现模块

### 功能说明

系统需要从长期记忆中发现重复出现的行为链路。

重点不是精确医学判断，而是帮助用户发现可行动的模式。

### 模式结构

一个行为模式由以下元素组成：

```text
场景 / 触发因素
↓
情绪反应
↓
应对行为
↓
结果
```

### 示例模式

```text
任务压力过大
↓
焦虑
↓
刷短视频
↓
学习中断、自责
```

```text
他人回复冷淡
↓
不安
↓
反复查看消息
↓
注意力下降
```

```text
项目进度落后
↓
自我否定
↓
逃避打开代码
↓
进度继续延迟
```

### 触发条件

MVP 阶段可以采用简单规则：

当相似模式出现次数大于等于 3 次，系统认为该模式值得提示用户。

例如：

```text
过去 14 天内，用户 3 次提到“任务压力 → 焦虑 → 回避学习”。
```

则生成模式发现提示。

### 输出示例

```text
我发现你最近几次学不进去时，前面通常都有一个共同点：任务看起来太大，导致你先焦虑，然后通过刷视频或拖延来暂时逃避，最后又开始自责。
```

---

## 8.6 干预策略库

### 功能说明

系统需要内置一组成长方法，用于将发现的问题转化为可执行行动。

MVP 阶段采用「方法库 + LLM 生成」的混合模式。

即：

1. 优先匹配内置方法
2. 匹配不到时由 LLM 根据上下文生成建议
3. LLM 生成的建议需要尽量具体、轻量、可执行

---

### 8.6.1 方法库示例

#### 拖延 / 难以开始

推荐方法：

```text
15 分钟启动法
```

使用场景：

* 用户知道该做什么，但不想开始
* 用户因为任务太大而焦虑
* 用户反复拖延学习、项目、简历、面试准备

行动建议：

```text
今天不要追求完成整个任务，只启动 15 分钟。
```

---

#### 内耗 / 反复想太多

推荐方法：

```text
认知记录表
```

使用场景：

* 用户反复纠结某件事
* 用户不断自我否定
* 用户情绪被某个人或某件事反复影响

行动建议：

```text
写下事件、想法、情绪、证据、替代解释和下一步行动。
```

---

#### 高敏感 / 过度关注外界评价

推荐方法：

```text
注意力回收训练
```

使用场景：

* 用户因为别人一句话反复难受
* 用户过度依赖别人的回复、认可、评价
* 用户情绪被外部反馈强烈影响

行动建议：

```text
减少不必要的信息接触，将注意力转向自己今天能完成的一件小事。
```

---

#### 长期迷茫 / 人生方向不清楚

推荐方法：

```text
人生意义探索写作
```

使用场景：

* 用户长期迷茫
* 用户不知道自己为什么努力
* 用户对赚钱、职业、人生状态感到困惑

行动建议：

```text
围绕“我想达到什么状态”“赚钱的目的是什么”“我如何反推当前行动”进行写作。
```

---

#### 持续焦虑、内耗、高敏感混合问题（P1）

推荐方法：

```text
18 天成长计划
```

使用场景：

* 用户长期被内耗、焦虑、高敏感影响
* 用户已经多次出现相似负面模式
* 用户愿意进入结构化成长计划

---

## 8.7 18 天成长计划模块（P1 设计参考）

本模块为 P1 结构化成长计划设计参考，MVP 不实现。

### 功能定位

18 天成长计划不是系统默认主流程，而是干预策略库中的一个结构化方案。

当系统发现用户存在持续性焦虑、内耗、高敏感、拖延等问题，并且用户愿意尝试结构化训练时，推荐进入 18 天计划。

---

### 计划结构

#### 第 1 天

用户完成两个任务：

1. 三格记录表
2. 选择一个需要提升的技能

三格记录表包括：

```text
今天做了什么
今天让自己不舒服的人和事
今天开心的三件小事
```

技能选择示例：

* 编程
* 英语
* 面试表达
* 写作
* 健身
* 项目开发

---

#### 第 2 天至第 10 天

每天重复完成：

1. 三格记录表
2. 专注学习技能 1.5 小时

系统每天记录：

* 完成情况
* 主要情绪
* 负面触发事件
* 开心小事来源
* 技能学习进度

---

#### 第 11 天

系统引导用户进行归纳总结。

总结维度包括：

1. 每天主要做了什么
2. 哪些事情反复让自己不舒服
3. 哪些人反复影响自己的情绪
4. 开心的小事更多来自外部评价，还是来自自我提升
5. 是否存在需要减少接触或切断联系的人和事

系统辅助生成阶段总结。

---

#### 第 12 天至第 15 天

主题：

```text
识别并减少消耗源
```

用户每天完成：

1. 分辨对自己不好的关系、事情和环境
2. 尝试减少非必要接触
3. 写 1 小时日记

系统重点关注：

* 用户是否能识别消耗源
* 用户是否采取行动减少接触
* 用户是否仍然反复关注外界评价

---

#### 第 16 天至第 18 天

主题：

```text
人生意义探索
```

用户每天围绕一个主题写作：

1. 我的人生想达到什么样的境界？
2. 我赚钱的目的是什么？
3. 我需要通过什么方式达到理想状态？

系统辅助用户总结：

* 长期目标
* 核心价值观
* 当前阶段最重要的行动方向

---

### 18 天计划产出

计划结束后，系统生成一份成长总结，包括：

```text
主要压力源
典型情绪模式
典型行为模式
外部评价依赖情况
自我提升来源
需要减少接触的人和事
长期人生目标
下一阶段行动建议
```

## 8.8 行动任务生成与跟踪

### 功能说明

系统不能只给建议，还需要将建议转化为可执行任务，并在后续对话中跟踪。

### 示例

系统发现：

```text
任务压力 → 焦虑 → 刷视频 → 自责
```

推荐：

```text
15 分钟启动法
```

生成任务：

```text
今天只完成 15 分钟的简历修改，不要求全部完成。
```

第二天跟踪：

```text
昨天那个 15 分钟简历修改任务完成了吗？
```

如果用户完成：

```text
记录完成情况，强化成功经验。
```

如果用户没完成：

```text
追问阻碍原因，并降低任务难度。
```

### 未完成任务处理

用户未完成任务时，Agent 不批评用户，而是分析阻碍：

* 是任务太大？
* 是时间不够？
* 是情绪阻力太强？
* 是不知道从哪开始？
* 是目标本身不重要？

然后重新调整任务。

## 8.9 成长总结模块（P1 设计参考）

### 功能说明

系统后续需要定期为用户生成成长总结。MVP 不实现完整周总结页面，只记录任务反馈和可用于后续总结的结构化数据。

P1 阶段支持：

* 周总结
* 18 天计划总结
* 阶段性手动总结

### 周总结内容

```text
本周主要压力源
本周高频情绪
本周重复行为模式
本周完成的行动
本周未完成的行动
下周建议
```

### 示例

```text
本周你最主要的压力源是秋招准备和项目推进。

你反复出现的模式是：

任务太大
↓
焦虑
↓
刷视频回避
↓
自责

本周你有两次成功完成了 15 分钟启动任务，这说明降低启动难度对你是有效的。

下周建议继续采用小任务启动，不要一开始就要求自己完成完整模块。
```

## 10. Agent 回复策略

Agent 需要根据不同场景选择不同回复策略。

### 10.1 情绪承接策略

适用于用户情绪明显但信息不足时。

目标：

* 让用户感到被理解
* 不急着分析
* 轻量追问一个关键问题

示例：

```text
听起来你最近不是单纯累，而是有点被压力压住了。最近有没有一件事让这种感觉特别明显？
```

---

### 10.2 信息补全策略

适用于记忆字段缺失时。

目标：

* 补全事件、情绪、触发、行为、结果
* 一次只问一个问题

示例：

```text
你说“学不进去”，更像是不知道从哪开始，还是开始后很快就想逃避？
```

---

### 10.3 模式反馈策略

适用于系统发现重复模式时。

目标：

* 温和反馈系统观察到的模式
* 让用户确认是否准确
* 不强行下结论

示例：

```text
我不确定这个判断是否准确，但从你最近几次描述看，好像经常是任务一变大，你就会先焦虑，然后通过刷视频暂时逃开，最后又开始自责。这个模式你觉得符合吗？
```

---

### 10.4 行动建议策略

适用于用户需要行动时。

目标：

* 不给大而空的建议
* 生成小任务
* 降低启动难度

示例：

```text
今天先不要求你完成整个项目模块，只做一件事：打开代码，花 15 分钟整理当前报错。做到这里就算完成。
```

---

### 10.5 复盘策略

适用于任务完成或未完成后。

目标：

* 完成时强化成功经验
* 未完成时分析阻碍
* 不批评用户

示例：

```text
没完成也没关系。我们先不评价你自律不自律，只看卡住的地方：是任务太大，还是你一想到要开始就有压力？
```

### 11.8 记忆与隐私治理

由于本产品的核心能力是长期记忆，MVP 必须提供最小数据治理能力：

1. **可见**：用户可以查看系统保存了哪些长期记忆。
2. **可改**：用户可以修改不准确的记忆字段。
3. **可删**：用户可以删除不希望保留的记忆。
4. **可追溯**：每条记忆需要保存来源，例如 conversation、checkin 或 reflection。
5. **可停用**：被用户拒绝的模式不得继续用于回复、任务生成或画像更新。

不应长期保存以下内容：

* 低置信度人格判断
* 医学诊断标签
* 与成长任务无关的普通闲聊
* 用户明确要求删除的信息
* 没有证据来源的敏感推断

## 14. 安全与边界

### 14.1 产品边界

系统必须明确声明：

```text
本产品用于自我成长记录和日常心理支持，不提供医学诊断、心理治疗或危机干预服务。
```

产品回复应避免：

* 给出医学诊断
* 暗示用户患有某种疾病
* 替代专业心理咨询
* 对高风险场景继续做普通成长建议
* 将一次性情绪写成稳定人格特征

### 14.2 风险等级

MVP 使用四级风险标记：

```text
none：普通成长对话
low：负面情绪明显，但无自伤、自杀、暴力意图
medium：出现强烈绝望、自我否定或模糊危险表达
high：出现自伤、自杀、伤害他人、明确危险计划或迫近风险
```

### 14.3 高风险内容处理

当用户表达以下内容时：

* 自伤
* 自杀
* 暴力
* 严重绝望
* 明确危险计划

系统不能继续普通成长教练流程，不能生成成长任务，也不能把该轮内容用于普通行为模式发现。

应该：

1. 表达关切
2. 建议立即联系可信任的人
3. 建议寻求专业帮助
4. 在必要时建议联系当地紧急救助服务
5. 只保存最小必要的安全处理记录，不沉淀为成长画像

MVP 阶段可先实现关键词和 LLM 双重识别。高风险召回率优先于精确率：宁可多触发一次安全回复，也不要漏掉明确危险表达。

### 14.4 误判恢复

如果用户说明自己并不存在危险，只是在夸张表达，系统可以回到普通成长教练流程，但应保持谨慎语气，并避免立刻给出强干预任务。

## 15. 评估指标

### 15.1 技术评估指标

用于项目展示和面试说明。MVP 建议准备 30 条人工标注测试输入，覆盖拖延、焦虑、内耗、高敏感、迷茫、普通闲聊和高风险表达。

#### 记忆抽取准确率

人工抽样评估情绪事件字段是否抽取正确。

字段包括：

* event
* emotion
* trigger
* behavior
* result

建议目标：核心字段 emotion、trigger、behavior 的人工通过率不低于 80%。

#### 追问合理性

评估系统是否只追问一个关键问题，且问题不冒犯、不像问卷、不强迫用户回答。

建议目标：单问题追问通过率不低于 90%。

#### 记忆召回相关性

评估 Agent 回复时召回的历史记忆是否与当前问题相关。

建议目标：Top 3 召回中至少 1 条相关记忆。

#### 模式发现准确率

人工判断系统发现的模式是否有足够证据支持。

建议目标：每个候选模式必须引用至少 3 条 evidence_memory_ids。

#### 干预匹配合理性

人工判断系统推荐的方法是否适合当前问题。

建议目标：干预方法必须低难度、具体、可在 15 到 30 分钟内启动。

#### 高风险识别召回率

人工测试高风险表达是否触发安全流程。

建议目标：明确自伤、自杀、暴力计划类输入必须进入 high 风险处理。

---

### 15.2 产品体验指标

MVP 阶段可记录：

* 用户连续使用天数
* 用户完成行动任务次数
* 用户确认模式准确的比例
* 用户修改或删除记忆的比例
* 用户对行动任务是否可执行的反馈

---

## 16. 项目成功标准

MVP 成功不以「用户焦虑明显降低」作为标准。

MVP 成功标准是：

1. 用户可以进行自由对话
2. 系统能从对话中抽取结构化记忆
3. 系统能主动追问缺失信息，且一次只问一个问题
4. 系统能召回相关长期记忆
5. 系统能基于至少 3 条证据发现一种候选行为模式
6. 系统能让用户确认或拒绝候选模式
7. 系统能根据已确认或高置信模式匹配干预方法
8. 系统能生成轻量行动任务
9. 系统能记录任务完成 / 未完成反馈
10. 用户可以查看、修改、删除长期记忆
11. 高风险输入会进入安全流程，不继续普通成长教练流程
12. 整体流程可以完整演示

## 17. 面试展示重点

项目展示时，不应强调：

```text
我做了一个心理聊天机器人。
```

而应强调：

```text
我做了一个面向长期成长场景的记忆驱动 Agent。
```

推荐介绍：

> 这个项目的核心不是让大模型陪用户聊天，而是解决长期陪伴 Agent 的三个关键问题：记忆从哪里来、记忆如何组织、记忆如何产生行动价值。系统通过主动记忆采集机制从自然对话中抽取事件、情绪、触发因素、行为和结果，再通过多层记忆架构沉淀长期用户画像，并基于重复模式发现和干预策略路由，将用户问题转化为可执行行动任务。

## 19. 总结

Memory-Driven Growth Agent 的核心不是心理咨询，而是长期陪伴 Agent 的记忆与行动闭环。

项目主线为：

```text
主动采集记忆
↓
分层管理记忆
↓
发现行为模式
↓
匹配成长方法
↓
生成行动任务
↓
跟踪执行反馈
```

该项目适合作为 AI Agent / LLM 应用开发方向的作品集项目，能够体现产品理解、Agent 架构设计、记忆系统设计和实际工程落地能力。
